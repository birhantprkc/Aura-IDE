## What this phase introduces

The user's mental model is: I chat with a *planner* (fast, cheap), and when an actual code change is needed, the planner produces a *spec* and dispatches it to a *worker* (heavier model, clean context, tool access). The worker reports back with a diff. The planner sees the result and surfaces it to the user. Conversation continues.

This mirrors how the user has been using me + claude-code-bridge — one model for planning, another for doing — except inside Aura, with their choice of models per role.

### Two roles, separate manager runs

- **Planner role**: V4 Flash by default. Has access to read-only tools: `read_file`, `list_directory`, `glob`. **Does not** have `write_file` / `edit_file`. Plus one new tool: `dispatch_to_worker`.
- **Worker role**: V4 Pro by default. Receives a spec (constructed by the planner via `dispatch_to_worker`). Has full tool access including `write_file` and `edit_file`. Runs in its own ConversationManager instance with a fresh history seeded from the spec. Its tool calls are subject to the same diff-approval flow as before.

When the planner calls `dispatch_to_worker`, the GUI:
1. Renders the spec as a card in the chat (collapsible, edit button)
2. Shows a "Dispatch" button on the spec card
3. User clicks Dispatch → worker spins up in a separate manager, streams its events into a sub-thread visually nested under the spec card
4. When worker completes, its final summary becomes a tool result that's fed back to the planner
5. Planner continues the conversation with awareness of what the worker did

### `dispatch_to_worker` tool spec

```python
{
  "type": "function",
  "function": {
    "name": "dispatch_to_worker",
    "description": "Dispatch a coding task to a worker model with file write access. Use this when the user has agreed to a code change and you have enough information to specify the change precisely. The worker has tools to read and edit files in the workspace. Provide a complete, self-contained spec — the worker does not see this conversation. Include: goal, files involved (use exact paths from your earlier read_file calls), the specific change to make, any constraints. The worker will return a summary of what it did.",
    "parameters": {
      "type": "object",
      "properties": {
        "goal": {"type": "string", "description": "One-sentence statement of what the change accomplishes."},
        "files": {"type": "array", "items": {"type": "string"}, "description": "Workspace-relative paths the worker should read and/or modify."},
        "spec": {"type": "string", "description": "Full prose specification of the change. Be specific. Reference function names, line behavior, error cases. The worker has not seen the conversation, so include necessary context."},
        "acceptance": {"type": "string", "description": "How the worker (and the user) knows the task is done. Concrete, checkable."}
      },
      "required": ["goal", "files", "spec", "acceptance"]
    }
  }
}
```

## Architecture changes

### `aura/conversation/manager.py`

Refactor `ConversationManager` so it can be instantiated for either role. Add a `role: str` field (`"planner"` or `"worker"`) and a `system_prompt` parameter. Different system prompts per role:

- **Planner system prompt** (rough — refine for tone): "You are the planner in Aura. The user chats with you to troubleshoot and modify their codebase. You have read-only file tools. When the user has agreed to a code change and you have enough information, call `dispatch_to_worker` with a complete spec. Do not propose code changes inline — propose them via dispatch. Before dispatching, confirm the change with the user in plain English. Keep your responses concise and oriented toward action."
- **Worker system prompt** (rough): "You are the worker in Aura. You execute a precise coding task specified by the planner. You have read/write file tools (edits gated by user approval via the GUI). Read the spec carefully, read the listed files, make the changes via `edit_file` or `write_file`, then return a brief summary of what you changed. Do not chat — your output is a structured report."

Don't intermingle the two roles in a single manager instance. The GUI will hold one planner manager (long-lived) and may hold zero or many worker managers (one per dispatch, short-lived).

### `aura/conversation/tools/registry.py`

Add a `read_only_tools_only: bool` constructor flag (or a method `set_read_only_for_planner()`) so the planner registry exposes only `read_file`, `list_directory`, `glob`, plus the new `dispatch_to_worker`. The worker registry exposes the existing five tools (read + write). Read-Only Mode at the toolbar level still works: if Read-Only is on, the worker registry also drops write tools (so dispatching becomes a no-op effectively, but the planner still functions for chat — and `dispatch_to_worker` should be removed from the planner's tools too in that case, since there's nothing for the worker to write).

`dispatch_to_worker` is implemented as a special tool whose `execute()` method **does not** run synchronously like file tools. Instead, it raises a sentinel result that the planner manager interprets as "pause this turn, hand control to the GUI to spawn a worker." The cleanest pattern: have `dispatch_to_worker.execute()` return a `WorkerDispatchRequest(goal, files, spec, acceptance)` object. The planner manager detects this in its tool loop and, instead of feeding the result back to the planner immediately, emits a new event `WorkerDispatchRequested(spec)` and **suspends** the planner loop until the GUI bridge resumes it with a worker result.

### `aura/bridge/qt_bridge.py`

Now manages two kinds of run:

- **Planner run**: long-lived background loop on a worker thread, same as today.
- **Worker dispatch**: when `WorkerDispatchRequested` event fires, the bridge:
  1. Emits a Qt signal so the chat view renders the spec card with Dispatch / Edit / Cancel buttons.
  2. Waits (blocking on a `QSemaphore` or futures-style result, same pattern as the diff-approval bridge) for the user to click Dispatch (or Cancel).
  3. On Dispatch: spawns a fresh `ConversationManager` configured as worker, runs it on a (separate) worker thread, streams its events into the chat view nested visually under the spec card.
  4. When the worker finishes, takes its final summary text + a structured "what happened" report (tool calls made, files modified, success/failure), packages that as the tool result for the original `dispatch_to_worker` call, resumes the planner loop with that result appended as the tool_result message.
  5. On Cancel: feeds back a tool result like `{"status": "cancelled", "reason": "user cancelled dispatch"}` to the planner so it knows.

The worker's diff-approval flow uses the **same** approval callback the GUI already provides — every write still goes through the diff dialog. The user is approving worker writes the same way they approve direct writes today.

### `aura/gui/chat_view.py`

New card type: `SpecCard`. Shows:
- Header: "Dispatch to Worker" with a small icon, accent-colored border-left
- Goal (one-line, bold)
- Files (chip list, monospace)
- Spec (collapsible — collapsed by default if longer than ~6 lines, with "Show more" affordance)
- Acceptance criteria (italic, muted)
- Buttons row: **Dispatch** (primary, accent), **Edit Spec** (secondary), **Cancel** (subtle)
- After dispatch: collapses into a header showing "Dispatched · <duration>" plus a nested area below that contains the worker's streaming output — its tool calls and content render as if it were a sub-conversation, indented and visually contained.

Edit Spec opens a small modal (new file `aura/gui/spec_edit_dialog.py`) with editable fields for goal/files/spec/acceptance. OK applies the edits to the dispatch_to_worker payload before the worker runs.

The worker's tool calls and content render in the same card style as planner tool calls but inside the spec card's nested area. Worker's final summary becomes the closing text of the spec card.

### `aura/gui/main_window.py`

Add to the toolbar (or settings):

- **Worker model picker** (Flash/Pro) — defaults to Pro
- **Planner thinking** toggle (Off/High/Max)

Existing single "Model" and "Thinking" pickers now control the planner specifically. Add the worker controls visibly. If toolbar gets crowded, group them: "Planner: [Flash] Thinking: [High]  •  Worker: [Pro] Thinking: [High]".

This is the "fallback" mode for sessions where the user doesn't want the split. When OFF, the worker controls in the toolbar are hidden/disabled.

## Definition of done

1. App launches in planner/worker mode by default. Toolbar shows planner+worker pickers.
2. Sending a chat message hits the planner only — planner can read files but cannot write.
3. Asking for a code change: planner produces a spec card with Dispatch/Edit/Cancel buttons. Cannot edit files directly.
4. Clicking Dispatch: worker spins up, its activity (tool calls, content) renders nested under the spec card, edits go through the diff approval flow.
5. Worker completes → planner continues the conversation with a confirmation message referencing what the worker did.
6. Settings toggle turns off planner/worker mode → app reverts to Phase 1+2 single-model behavior. Smokes for the old path still pass.
7. Conversation save/load round-trips planner-worker conversations including any dispatches.
8. `scripts/smoke_planner_worker.py` passes end-to-end with auto-approve callbacks.
9. No regressions: chat still streams, status bar still updates, Read-Only Mode still works (and disables dispatch when on).

## Tone considerations

The planner's system prompt should make it actually behave like a planner — it should ask clarifying questions before dispatching, not jump straight to dispatch on the first user message. Tune the prompt: "Before calling `dispatch_to_worker`, you should have read the relevant files and confirmed the user's intent. If the user's request is ambiguous, ask. If files might be involved that you haven't read, read them first."

The worker's prompt: tight, mechanical. It does the job and reports. No conversation.
